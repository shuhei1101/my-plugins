# worktree — Git Worktree実装ワークフロー管理スキル

## このスキルの目的

Git worktreeを使って、複数のAIセッションが干渉せず並行して実装できるワークフローを提供する。

1セッション = 1PRの単位で、ブランチ作成 → worktree隔離 → 実装計画（PRドキュメント）→ 実装 → レビュー → squashマージ → クリーンアップ、までを対話形式で進める。

## 前提知識

以下のファイルを参照すること:

- `references/workflow_guide.md` — 対話形式スキルの基本原則、選択肢ルール、進捗バー、多角度質問の提示形式、「質問シート」の定義
- `references/pr_doc_template.md` — `docs/PR/PR{N}.md` のテンプレート
- `references/git_commands.md` — git worktree / squash merge の具体コマンド

## Memory活用ルール

- セッション記録: `~/.claude/skill-memory/worktree/{YYYYMMDDHHMMSS}_session.md`
- セッション開始時に作成し、ベースブランチ・worktreeパス・PR番号・現在のステップを記録
- セッション中断時の再開、および忘却防止のため、各フェーズ完了時に更新する

---

## フェーズ1: 実装計画

### ステップ1.1(モード選択)

**条件**: スキル起動時

**入力**: ユーザーのリクエスト全文

**AI実施タスク**:
- 起動時のコンテキスト分析:
  - 「PR再開」「前回の続き」等 → 既存worktreeの作業再開モード（該当セッションのステップへジャンプ）
  - 「worktree一覧」等 → worktree一覧表示
  - それ以外 → 通常の選択肢提示
- `~/.claude/skill-memory/worktree/{YYYYMMDDHHMMSS}_session.md` を作成しセッション開始を記録
- `git worktree list` で既存worktree一覧を取得し、関連するものがあれば表示

**出力**:
- スキルの説明
- 既存worktree一覧（あれば）

**選択肢**:
1. 新規PRを作成する → ステップ1.2へ
2. 既存worktreeの作業を再開する（worktreeパス or PR番号指定）→ skill-memoryからセッション復元、進行中のステップへ
3. worktree一覧のみ確認する
4. その他（入力してください）

**補足**:
- skill-memoryのセッションファイルから、`## 現在のステータス` を読み取って該当ステップに復帰できるようにする
- 実装・ドキュメント更新問わず、すべての作業で worktree を作成する（ブランチのみモードは廃止。複数 AI セッション間の干渉を防ぐため）

---

### ステップ1.2(実装内容のヒアリング)

**条件**: ステップ1.1で「新規PRを作成する」が選択された場合

**入力**: -

**AI実施タスク**:
- プロジェクトのコンテキストを把握（README、docs配下等を軽くスキャン）してヒアリング文面を調整
- ユーザーが冒頭のリクエストですでに実装内容を伝えている場合は、その内容を要約して提示

**出力**:
- 「今回のPRで実装したい内容を教えてください。概要・対象範囲・やりたいこと・気になる点など、自由にお話しください。」
- （すでに伝わっている場合）「冒頭で伺った内容を整理すると `XXX` です。追加・修正したい点があれば教えてください。」

**選択肢**: なし（自由入力）

**補足**:
- このステップでは選択肢を出さない
- ユーザーの回答は次のステップ（1.3）の入力となる

---

### ステップ1.3(実装内容の整理とPRドキュメント作成)

**条件**: ステップ1.2でユーザーが実装内容を伝えた後

**入力**: ユーザーが伝えた実装内容

**AI実施タスク**:
- 実装内容から不明点・確認事項を抽出
  - `/idea` を使用して複数観点から質問を生成（実装者目線・ユーザー目線・エッジケース目線）
  - 不明点があれば質問シート（Q1/Q2/Q3...）として整理（`references/workflow_guide.md` の多角度質問の提示形式に従う）
- 不明点がない、または質問シート回答後:
  - `docs/PR/` 配下の既存ファイルをスキャンして次のPR番号を決定（`docs/` が無ければ作成）
  - `docs/PR/PR{N}.md` の内容を作成（`references/pr_doc_template.md` のテンプレート適用）

**出力**:
- 不明点がある場合: 質問シート + `💡 Tips: /copy コマンドを使用してください`
- 不明点がない場合: 作成予定のPRドキュメント内容プレビュー + PR番号

**選択肢**:
- 質問シート提示時: なし（自由入力）
- プレビュー提示時:
  1. この内容でPRドキュメントを作成する → 次のステップ（1.4）へ
  2. 内容を修正したい（伝えてください）
  3. 質問シートで追加ヒアリングしたい
  4. その他（入力してください）

---

### ステップ1.4(PRドキュメントのユーザー確認ループ)

**条件**: ステップ1.3でPRドキュメントのプレビューが承認された後

**入力**: 作成済み（または作成予定）の `docs/PR/PR{N}.md` の内容

**AI実施タスク**:
- `docs/PR/PR{N}.md` をまだ作成していなければこのタイミングで作成
- ユーザーが修正指示を出した場合はファイルを更新して差分を提示

**出力**:
- 現在のPRドキュメント内容（概要・作業内容・実装・テスト）を整形して表示
- 「この内容で実装に進んでOKですか？気になる点があれば質問シートで追加ヒアリングもできます」

**選択肢**:
1. この内容で確定 → フェーズ2（準備）へ
2. 修正したい点を伝える → 修正してこのステップを再実行
3. 質問シートで追加ヒアリングしたい → 質問シート生成後このステップを再実行
4. やっぱり実装しない（中止） → セッション終了（skill-memoryに「中止」を記録）
5. その他（入力してください）

**補足**:
- 「中止」が選ばれた場合、worktreeはまだ作成していないので安全に終了可能
- PRドキュメントは残す／削除はユーザーに確認

---

## フェーズ2: 準備

### ステップ2.1(ベースブランチ確認)

**条件**: フェーズ1の確認ループを抜けた後

**入力**: -

**AI実施タスク**:
- `git branch --show-current` でカレントブランチを取得
- `git status` で作業ツリーがクリーンかチェック
- ベースブランチ情報を skill-memoryセッションファイルに保存

**出力**:
- 「現在のブランチは `{ブランチ名}` です。このブランチから新しいブランチを切って作業を開始します」
- カレントブランチが `master`/`main` の場合、注意喚起
- 未コミット変更がある場合は警告

**選択肢**:
1. このブランチから切ってOK → 次のステップ（2.2）へ
2. 別のブランチをベースにしたい（ブランチ名を指定）→ `git checkout` してこのステップを再実行
3. 未コミット変更を先にコミット/stashしたい → ユーザー作業待ち
4. 中止する
5. その他（入力してください）

**補足**:
- ベースブランチは「最終的にsquashマージする戻り先」なので確実に確認する
- skill-memoryに保存することで、マージフェーズで正しい戻り先を参照できる

---

### ステップ2.2(ブランチ名決定)

**条件**: ステップ2.1でベースブランチが確定した後

**入力**: PR番号、ユーザーからの「やること名」

**AI実施タスク**:
- PRドキュメントの `## 概要` から「やること名」の候補を推論して提案
- ユーザーが名前を指定したら `{PR番号}/{やること名}` 形式でブランチ名を組み立てる
  - スペース/特殊文字はハイフン・アンダースコアに変換
  - 日本語はそのまま許容
- `git branch --list {ブランチ名}` で衝突チェック

**出力**:
- 「ブランチ名を決めます。PR番号は `{N}` です」
- 候補: 「やること名の候補として `{候補1}` `{候補2}` `{候補3}` を考えました」
- 完成形: `{PR番号}/{やること名}`（例: `30/ログイン実装`）

**選択肢**:
1. 提案された名前で確定 → 次のステップ（2.3）へ
2. 自由入力でやること名を指定 → 再提示
3. PR番号を変更したい（手動指定）
4. 戻る
5. その他（入力してください）

---

### ステップ2.3(ブランチ作成・Worktree作成・空コミット)

**条件**: ステップ2.2でブランチ名が確定した後

**入力**: ブランチ名、ベースブランチ、PR番号

**AI実施タスク**:
- 実行前にユーザーへ実行内容を提示して確認
- 確認後、以下を順に実行（詳細コマンドは `references/git_commands.md` 参照）:
  1. `git branch {ブランチ名} {ベースブランチ}`
  2. worktreeパスを決定: デフォルト `{親ディレクトリ}/{リポジトリ名}-wt-PR{N}`
  3. `git worktree add {worktreeパス} {ブランチ名}`
  4. **プロジェクトタイプを自動検出してシンボリックリンクを作成**:
     - `pyproject.toml` または `setup.py` が存在する → Python プロジェクト
       - メインリポジトリの `venv/` が存在すれば `ln -s {メインリポジトリ}/venv {worktreeパス}/venv`
     - `package.json` が存在する → Node.js/Next.js プロジェクト
       - `node_modules/` が存在すれば `ln -s {メインリポジトリ}/node_modules {worktreeパス}/node_modules`
       - `.next/` が存在すれば `ln -s {メインリポジトリ}/.next {worktreeパス}/.next`
     - リンク対象が存在しない場合はスキップ（警告のみ表示）
  5. worktree内で空コミット: `git commit --allow-empty -m "chore: start PR{N} {やること名}"`
  6. worktree内でPRドキュメントをコミット: `git add docs/PR/PR{N}.md && git commit -m "docs: add PR{N} plan"`
- skill-memoryに worktreeパス・ブランチ名・PR番号を保存

**出力**:
- 実行予定コマンドのプレビュー
- 「以下を実行します。問題なければ選択してください」

**選択肢**:
1. 実行する → コマンド実行後、次のステップ（3.1）へ
2. worktreeパスを変更したい（入力）→ このステップを再実行
3. 中止する
4. その他（入力してください）

---

## フェーズ3: 実装

### ステップ3.1(実装スキル指定)

**条件**: ステップ2.3完了後

**入力**: -

**AI実施タスク**:
- 現在利用可能なスキル一覧から、実装に使えそうなものをピックアップして提案
  - 例: `py-wizard`, プロジェクト固有のスキルなど

**出力**:
- 「実装に使用するスキルを指定してください」
- 推奨スキル一覧（プロジェクトの性質から判断した候補）

**選択肢**:
1. 推奨スキルから選択（番号指定）→ 次のステップ（3.2）へ
2. 自由入力でスキル名を指定
3. スキルを使わず直接実装する
4. その他（入力してください）

---

### ステップ3.2(実装実行)

**条件**: ステップ3.1で実装方針が決まった後

**入力**: 実装スキル名（または「直接実装」）、PRドキュメント

**AI実施タスク**:
- **基本方針**: worktreeパスで作業し、本スキルと実装スキルを両方コンテキストに読み込む
  - 両方読み込み可能な場合: 本スキルの進行状況を把握しつつ、実装スキルのステップに沿って実装
  - コンテキストが肥大化する懸念がある場合: サブエージェントに実装スキルとPRドキュメントを渡して実装させる
- 実装中は skill-memory の `## 現在のステータス` を定期的に更新
- `docs/PR/PR{N}.md` の `## 作業内容` タスクリストを進捗に合わせて更新
- 実装スキルが対話形式の場合、そのスキルのフローをそのまま進める
- 実装完了後、変更されたファイルを確認

**出力**:
- 実装の進捗状況
- 完了時: 変更ファイル一覧と `## 作業内容` チェック状況

**選択肢**:
1. 実装が完了した → 次のステップ（3.3）コミットへ
2. 追加で実装したい内容を伝える → 実装継続
3. 実装スキルを変更したい → ステップ3.1に戻る
4. 中止する（worktreeは残す）
5. その他（入力してください）

**補足**:
- worktreeパスで作業すること。メインリポジトリ側では作業しない
- 複数セッション並行時は、進捗が skill-memory に残っているので他セッション干渉を避けられる

---

### ステップ3.3(コミット)

**条件**: ステップ3.2で実装が完了した後

**入力**: 変更ファイル一覧、実装内容サマリ

**AI実施タスク**:
- コミットメッセージ生成（Conventional Commits形式）:
  - プレフィックス: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:` 等から適切なものを選択
  - 本文: 変更内容を簡潔に
- ユーザー確認後、worktree内で `git add . && git commit -m "{メッセージ}"` を実行
- コミット後、worktreeパスをユーザーに伝え、レビューを依頼

**出力**:
- コミットメッセージのプレビュー（プレフィックス付き）
- 「このメッセージでコミットします」

**選択肢**:
1. このメッセージでコミットする → 次のステップ（4.1）へ
2. メッセージを修正する（自由入力）
3. コミットを複数に分けたい → 分割指示を受けて再実行
4. その他（入力してください）

---

## フェーズ4: レビュー & マージ

### ステップ4.1(ユーザーレビュー)

**条件**: ステップ3.3でコミット完了後

**入力**: worktreeパス、コミットハッシュ

**AI実施タスク**:
- worktreeパスとコミット情報を明示
- ユーザーに「worktree内で実装内容を確認してください」と伝える
- レビュー結果を待つ

**出力**:
- 「以下のworktreeで実装内容を確認してください: `{worktreeパス}`」
- 「問題なければマージフェーズに進みます。修正が必要なら戻って実装を続けます」

**選択肢**:
1. レビューOK → 次のステップ（4.2）マージへ
2. 修正が必要 → ステップ3.2（実装実行）に戻る
3. 修正が必要（小さな修正、そのままコミット追加）→ ステップ3.2に戻って追加実装
4. 中止する（worktreeは残す／削除の確認へ）
5. その他（入力してください）

**補足**:
- このステップでユーザーは別の端末/エディタでworktreeを開いて実機確認することが想定される

---

### ステップ4.2(Squashマージ)

**条件**: ステップ4.1でレビューOKを得た後

**入力**: ベースブランチ、作業ブランチ、PR番号、機能名

**AI実施タスク**:
- squashマージメッセージを生成: `{機能名}の実装 #PR{N}` または `{機能名}のバグ修正 #PR{N}` 等
- 実行予定コマンドをユーザーに提示し、**ユーザーの「実行する」選択を待つ**

**出力**:
- マージメッセージのプレビュー
- 実行予定コマンドリスト:
  ```
  git checkout {ベースブランチ}
  git merge --squash {作業ブランチ}
  git commit -m "{機能名}の実装 #PR{N}"
  ```
- 「ベースブランチ `{ベースブランチ}` にsquashマージします。実行してよいですか？」

**選択肢**:
1. 実行する → コマンド実行後、次のステップ（4.3）へ
2. メッセージを修正する
3. 中止する
4. その他（入力してください）

**⚠️ 重要**: main/master へのマージはユーザーが「実行する」を選択するまで絶対に実行しないこと。

**補足**:
- リモートへのpushは自動で行わない（ユーザーが手動で行う）
- マージ後の push や PR作成はユーザー責任

---

### ステップ4.3(クリーンアップ確認)

**条件**: ステップ4.2でsquashマージ完了後

**入力**: worktreeパス、ブランチ名

**AI実施タスク**:
- 削除予定のworktreeパスとブランチ名を提示
- 未コミット変更や未マージコミットがないか最終確認

**出力**:
- 「以下を削除します」
  - Worktree: `{worktreeパス}`
  - ブランチ: `{ブランチ名}`
- 「問題なければ削除します」

**選択肢**:
1. 問題なし、削除する → 次のステップ（4.4）へ
2. worktreeだけ残したい → ブランチのみ削除
3. ブランチだけ残したい → worktreeのみ削除
4. 両方残す（このPR作業は一旦保留）→ セッション終了（memoryに「保留」を記録）
5. その他（入力してください）

---

### ステップ4.4(Worktree削除 & ブランチ削除)

**条件**: ステップ4.3で削除が承認された後

**入力**: worktreeパス、ブランチ名

**AI実施タスク**:
- 実行コマンド（詳細は `references/git_commands.md`）:
  1. `git worktree remove {worktreeパス}`
  2. `git branch -D {ブランチ名}`
- skill-memoryセッションファイルを更新（`## 現在のステータス: 完了`）
- 完了メッセージを表示

**出力**:
- 実行結果
- 「PR{N} の作業が完了しました。お疲れ様でした！」
- 次回のPRを作るかどうかの案内

**選択肢**:
1. 完了（セッション終了）
2. 続けて新しいPRを作成する → ステップ1.2へ
3. その他（入力してください）

**補足**:
- リモートpushはユーザー責任のため、このスキルでは扱わない
- skill-memoryは履歴として残す（次回スキル起動時に参照可能）
