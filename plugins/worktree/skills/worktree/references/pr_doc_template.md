# PRドキュメントテンプレート

`docs/PR/PR{N}.md` のフォーマット。

## テンプレート

```markdown
## 概要
{単語レベルで簡潔に何をするか}

## 作業内容
- [ ] {タスク1}
- [ ] {タスク2}
- [ ] {タスク3}

## 実装
| 追加/編集 | ファイルパス | クラス.メソッド | 変更内容 |
|-----------|-------------|----------------|---------|
| 追加 | src/foo.py | Foo.bar | 新規メソッド追加 |
|  |  | Foo.baz | ヘルパー追加 |
| 編集 | src/main.py | main | Foo.barの呼び出し追加 |

## テスト
| 追加/編集 | ファイルパス | テスト対象ファイルパス | クラス.メソッド | 変更内容 |
|-----------|-------------|----------------------|----------------|---------|
| 追加 | tests/test_foo.py | src/foo.py | TestFoo.test_bar | barメソッドのテスト |
|  |  |  | TestFoo.test_baz | bazメソッドのテスト |
```

## セクションの書き方ルール

### 概要
- 単語レベルで簡潔に
- 不要な装飾文は入れない

### 作業内容
- Markdownのタスクリスト（`- [ ]`）のみ使用
- 不要な文章は入れない
- 実装が進むにつれてチェックを入れていく

### 実装
- カラム順: `追加/編集` | `ファイルパス` | `クラス.メソッド` | `変更内容`
- 同じファイルに複数メソッドを追加する場合、ファイルパスは最初の行だけに書き、後続の行では空欄にする
- クラスがある場合は `クラス名.メソッド名` 形式

### テスト
- 実装と同じ構造 + `テスト対象ファイルパス` カラム追加
- カラム順: `追加/編集` | `ファイルパス` | `テスト対象ファイルパス` | `クラス.メソッド` | `変更内容`
- 重複するファイルパスは省略ルール同様

## 追加セクション

必要に応じて以下のセクションを追加してよい:
- `## 設計メモ` — 設計判断の記録
- `## 依存関係` — 他PRや外部ライブラリへの依存
- `## リスク` — 懸念事項
