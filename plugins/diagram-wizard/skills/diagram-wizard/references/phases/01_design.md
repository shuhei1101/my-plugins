# フェーズ2: 設計

---

## ステップ2.1（粒度の確認）

**条件**: ステップ1.2で図の対象情報が確定した後

**入力**: ユーザーの粒度に関する希望（自然言語で自由入力）。例:
- 「大・中・小で分けて」
- 「概要レベルでOK」
- 「詳細まで全部入れて」
- 「まず全体像を見せて、詳細は後で」
- 入力なしの場合はAIが推奨を提示

**AI実施タスク**:
- 対象の規模・複雑度を分析し、適切な粒度レベルを推奨する
- 各粒度レベルについて**具体的な収録内容の例**を提示する（抽象的な説明だけでなく、実際に何が含まれるかをイメージできるように）
- 推奨粒度レベルの数は対象の規模に応じて調整する:
  - **小規模な対象**（READMEの図、単一クラスのフロー等）: 単一レベルのみ推奨。「この規模であれば1枚の図で十分です」と伝える
  - **中規模な対象**（1モジュール、数クラスの連携等）: 2段階（概要/詳細）を推奨
  - **大規模な対象**（プロジェクト全体、複数モジュール連携等）: 2〜3段階を推奨
- **要素数の見積もりと警告**:
  - 各粒度レベルで図に含まれるおおよその要素数（ノード数、接続数）を見積もる
  - 1枚の図に要素が多すぎる場合（目安: mermaidでノード20個超、接続30本超）は明確に警告する
  - 警告例: 「この粒度では1枚の図に約35個のノードが含まれる見込みです。mermaidにはズーム機能がないため、可読性が著しく低下します。セクション分割または粒度の粗め調整を推奨します。」
  - 分割案を具体的に提示する（例: 「モジュールAとモジュールBで分割」「レイヤーごとに分割」等）

**出力**:
- 粒度レベルの推奨案（テーブル or リスト形式）

例（大規模プロジェクトの場合）:
```
📐 粒度レベルの推奨

対象の規模: 大（モジュール5個、クラス約30個）

| レベル | 粒度   | 収録内容の例                         | 推定要素数 |
|--------|--------|--------------------------------------|------------|
| Lv.1   | 粗め   | モジュール間の連携・依存関係         | ~8ノード   |
| Lv.2   | 中     | 各モジュール内の主要クラスと関係     | ~15ノード/図 |
| Lv.3   | 細かい | 各クラスの属性・メソッド・詳細フロー | ~10ノード/図 |

💡 推奨: Lv.1 + Lv.2 の2段階構成
   （Lv.3はクラス図として別途作成するのが効果的です）

⚠️ Lv.2を1枚にまとめると約30ノードになるため、
   モジュールごとに分割して作成することを推奨します。
```

例（小規模な対象の場合）:
```
📐 粒度レベルの推奨

対象の規模: 小（関数3つの処理フロー）

この規模であれば1枚の図で全体を表現できます。
推定要素数: ~6ノード

分割の必要はありません。
```

**選択肢**:
1. 推奨案でOK → ステップ2.2へ
2. 粒度を変更する（自由入力で指定） → 再分析してこのステップを再実行
3. その他（入力してください）

**補足**:
- ユーザーが最初から粒度を明示している場合（例: 「概要だけでいい」）、推奨案の提示は簡潔にし、確認のみ行う
- 要素数の見積もりはあくまで目安であり、正確な数はステップ2.2の構成提案で確定する
- mermaidの可読性限界を常に意識する。美しい図 > 情報量の多い図

## ステップ2.2（構成の提案）

**条件**: ステップ2.1で粒度が確定した後

**入力**: -

**AI実施タスク**:
- 確定した粒度に基づき、図の具体的な構成要素を洗い出す
- ファイル/コードが提供されている場合はその内容を分析して構成要素を抽出する
- 自然言語の説明のみの場合はその内容から構成要素を推定する
- 構成要素をテーブル or リスト形式で整理する
- **可読性チェック**:
  - 1枚の図に含まれる要素数を確認する
  - 要素数が多すぎる場合はセクション分割案を提示する
  - 分割する場合: 各セクション（図）のタイトル・含まれる要素・他セクションとの接点を明示する
- 図の種類に応じた構成要素の整理:
  - **フロー図**: ステップ名、条件分岐、開始/終了ポイント
  - **クラス図**: クラス名、主要属性/メソッド、クラス間の関係（継承/実装/依存/集約）
  - **シーケンス図**: アクター/参加者、メッセージの流れ、ループ/条件分岐
  - **ER図**: エンティティ名、主要属性、リレーション（1:1, 1:N, N:M）
  - **ユースケース図**: アクター、ユースケース、関係（include/extend）
  - **フォルダ構成図**: ディレクトリ構造、各フォルダ/ファイルの役割説明
  - **アスキーアート**: レイアウト構成、主要パーツの配置

**出力**:
- 図の構成案（テーブル or リスト形式）

例（クラス図の場合）:
```
📋 構成案: クラス図（Lv.2 — モジュールA）

| # | 要素          | 種類   | 主要な属性/メソッド              | 関係                        |
|---|---------------|--------|----------------------------------|-----------------------------|
| 1 | BaseHandler   | 抽象   | handle(), validate()             | -                           |
| 2 | FileHandler   | クラス | file_path, read(), write()       | BaseHandler を継承          |
| 3 | ApiHandler    | クラス | endpoint, request(), parse()     | BaseHandler を継承          |
| 4 | Config        | クラス | settings, load(), get()          | FileHandler, ApiHandler が依存 |
| 5 | Logger        | クラス | level, log(), rotate()           | 全クラスが依存              |

合計: 5ノード、7接続 ✅ 可読性OK
```

例（要素数超過で分割を提案する場合）:
```
📋 構成案: フロー図

⚠️ 全体で28ステップ・12分岐が検出されました。
   1枚の図では可読性が低下するため、以下の3セクションに分割を提案します:

【セクション1: 入力処理フロー】（9ステップ）
  - ファイル読み込み → バリデーション → パース
  - 接続先: セクション2の「データ変換」へ

【セクション2: データ変換フロー】（11ステップ）
  - データ変換 → フィルタリング → マッピング
  - 接続元: セクション1、接続先: セクション3

【セクション3: 出力処理フロー】（8ステップ）
  - フォーマット → 出力 → ログ記録
  - 接続元: セクション2
```

**選択肢**:
1. この構成でOK → ステップ2.3へ
2. 要素を追加・削除・修正する（自由入力で指定） → 修正してこのステップを再実行
3. 分割方法を変更する（分割提案がある場合のみ表示） → 再分割してこのステップを再実行
4. 粒度を見直す → ステップ2.1へ戻る
5. その他（入力してください）

**補足**:
- 構成案はあくまで「図に含める要素の一覧」であり、レイアウトやmermaidコードはこの段階では生成しない
- ユーザーが要素の追加・削除・修正を指示した場合は、変更を反映して再提示する
- 分割セクション間の接続点（どこからどこへつながるか）を必ず明示し、全体像を見失わないようにする

## ステップ2.3（構成の確認）

**条件**: ステップ2.2で構成案が提示された後

**入力**: ユーザーの確認結果（OK / 修正指示）

**AI実施タスク**:
- ユーザーの回答を分析する
- OKの場合: 構成を確定し、次のフェーズ（図の生成）へ進む準備をする
- 修正指示の場合:
  - 指示内容を解釈し、構成案に反映する
  - 修正後の構成案を再提示する（ステップ2.2へループバック）
  - 変更箇所をハイライトして差分がわかるようにする（例: 追加は `[+]`、削除は `[-]`、変更は `[~]`）
- 確定した構成の最終サマリーを表示する

**出力**:
- 修正がない場合:
```
✅ 構成が確定しました。

【確定サマリー】
- 図の種類: クラス図
- 粒度: Lv.2（モジュール内の主要クラス）
- セクション数: 2（モジュールA、モジュールB）
- 総要素数: 12ノード

次のステップで図を生成します。
```
- 修正がある場合: ステップ2.2の出力形式で再提示（変更箇所にマーク付き）

**選択肢**:

修正なしの場合:
1. 図の生成に進む → 次フェーズ（生成）へ
2. やっぱり修正する → 修正内容を自由入力 → ステップ2.2へ
3. 粒度から見直す → ステップ2.1へ
4. その他（入力してください）

修正ありの場合（ステップ2.2へループバック）:
- ステップ2.2の選択肢を再提示

**補足**:
- このステップはステップ2.2との間でループする確認ステップであり、ユーザーが納得するまで何度でも修正可能
- ループ回数が多い場合（3回以上）は「構成の方向性自体を見直しますか？」と提案してもよい
- 確定時にセッション情報（図の種類、粒度、構成要素一覧）をメモリに保存する
