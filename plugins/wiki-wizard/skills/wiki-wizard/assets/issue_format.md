# Issue 記載フォーマット

`docs/Issues.md` に追記する Issue のフォーマット定義。

---

## 基本フォーマット

```markdown
## ISSUE-{番号}: {タイトル} {未決定 | 検討中 | 保留}

**背景**: {なぜこのイシューが必要か。1〜3 行で簡潔に}

**選択肢**:
- **A**: {選択肢 A の内容と特徴}
- **B**: {選択肢 B の内容と特徴}
- **C**: {選択肢 C の内容と特徴（任意、2案でもよい）}

**推奨方式**: {A/B/C} — {推奨理由（AI が検討した根拠付き提案）}

---
```

---

## サブ Issue フォーマット（複数の Q がある場合）

```markdown
## ISSUE-{番号}: {大テーマのタイトル} {未決定 | 検討中 | 保留}

**背景**: {全体の背景}

---

### ISSUE-{番号}-1: {サブ質問1}

**背景**: {サブ質問1 の背景}

**選択肢**:
- **A**: ...
- **B**: ...

**推奨方式**: {A/B} — {理由}

---

### ISSUE-{番号}-2: {サブ質問2}

**背景**: {サブ質問2 の背景}

**選択肢**:
- **A**: ...
- **B**: ...

**推奨方式**: {A/B} — {理由}

---
```

---

## ステータスの使い分け

| ステータス | 意味 |
| --- | --- |
| `未決定` | まだ何も議論していない。最初に追加した状態 |
| `検討中` | ユーザーと議論中。選択肢を絞っている段階 |
| `保留` | 今は決定できない。外部依存・情報待ちなど |

## 良い Issue の書き方

### ❌ 悪い例 — 曖昧で選択しづらい

```markdown
## ISSUE-001: エラー処理について 未決定

**背景**: エラーが起きたときの処理が決まっていない。
```

### ✅ 良い例 — 選択肢と推奨方式が明確

```markdown
## ISSUE-001: STT 連続失敗時のフォールバック動作 未決定

**背景**: 音声認識（STT）が連続して失敗した場合、配信を止めるか継続するかが未定義。
失敗時に配信が完全停止すると視聴者体験が悪化するため、フォールバック方針が必要。

**選択肢**:
- **A**: 即座に Standby シーンへ遷移（シンプル、確実だが軽微な失敗でも止まる）
- **B**: 3 回連続失敗で Standby 遷移（ある程度の耐性あり）
- **C**: 失敗回数に応じて段階対応（複雑だが柔軟）

**推奨方式**: B — 1〜2 回の一時的な失敗に耐性を持ちつつ、連続失敗で確実に安全状態に移行できる。
設定可能化（起動時設定で閾値を変更可）にすれば柔軟性も確保できる。

---
```

## 番号採番ルール

- 連番: `ISSUE-001`, `ISSUE-002`, ...（3桁固定でゼロパディング）
- 採番手順:
  1. `docs/Issues.md` を Grep して最大番号を取得
  2. `docs/イシュー履歴.md` を Grep して最大番号を取得
  3. 両方の最大値 + 1 を新しい番号にする
- 一度発行した番号は使い回さない（削除された Issue の番号も予約済みとして扱う）
