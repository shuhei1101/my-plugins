# wiki-wizard 共通ルール

このドキュメントは wiki-wizard 全フェーズで共通の運用ルールを定義する。

---

## 📚 ドキュメント記載時の重複チェックルール

**目的**: ドキュメント間で情報が重複すると古い情報が残るリスクが高まるため、記載前に必ず重複チェックを行う。

### 記載前のチェック手順

新しい情報を書く前に、以下を必ず確認する:

1. **同じ情報が他のドキュメントにすでに書かれていないか？**
   - Grep ツールで該当キーワードを `docs/` 配下で全文検索
   - ある場合 → そのドキュメントが「マスター」かを判断（後述の原則）

2. **マスタードキュメントの判定原則**:
   - **機能仕様** → 機能別ドキュメント（例: `YouTube配信モード.md`）がマスター
   - **全モード共通仕様** → `共通仕様.md` がマスター
   - **未決定事項・残課題** → `Issues.md` がマスター
   - **決定済みの設計判断（経緯含む）** → 機能別ドキュメントがマスター（`Issues.md` には残さない）
   - **API 仕様** → 各 API ドキュメントがマスター
   - **AI 構成・LLM 役割分担・メッセージ構造** → `AI構成.md` 等の専用ドキュメントがマスター（プロジェクトに該当ドキュメントがある場合）

3. **マスター以外への記載は「参照リンク」に統一**:
   - 詳細をマスターに集約し、他のドキュメントは概要 + 参照リンクのみ
   - 例: `**XXX について**: 詳細は [マスター.md - 該当セクション](マスター.md#該当セクション) を参照`

---

### 記載例

**❌ 悪い例** — 同じ詳細仕様が 2 箇所に書かれている:

```markdown
# 共通仕様.md
## 連続失敗閾値
- STT: 3 回, TTS: 2 回, LLM: 2 回

# YouTube配信モード.md
## 連続失敗閾値
- STT: 3 回, TTS: 2 回, LLM: 2 回   ← 重複！
```

**✅ 良い例** — 詳細はマスターに集約、他は参照:

```markdown
# 共通仕様.md（マスター）
## 連続失敗閾値
- STT: 3 回, TTS: 2 回, LLM: 2 回
- 設定可能化（起動時設定画面で項目別に指定）

# YouTube配信モード.md
## エラー時の動作
... 詳細な閾値は [共通仕様.md - 連続失敗閾値](共通仕様.md#連続失敗閾値) を参照
```

---

### 既存ドキュメントの整合性維持

- ドキュメント更新時、関連する他のドキュメントに古い情報がないか確認する
- 仕様が変わったら、参照リンク先（マスター）を更新するだけで他は自動的に最新になる
- マスター変更時は、そのドキュメント名・アンカーを参照しているドキュメントを Grep でチェックする

### 参照リンクの書き方

- 同じディレクトリ内: `[ドキュメント名](ファイル名.md)`
- 特定セクションへ: `[ドキュメント名 - セクション名](ファイル名.md#セクションの-id)`
  - セクション ID は Markdown が自動生成（日本語混在時は注意、URL エンコードが必要なら手動で対応）

## 📝 最終更新の記録ルール

ドキュメントを更新した際は、末尾の `**最終更新**` 行に **変更内容の概要（1行）** を必ず付記する。

### 書き方

```markdown
**最終更新**: YYYY-MM-DD — {変更内容の概要（何を追加・変更・削除したか）}
```

### 具体例

```markdown
**最終更新**: 2026-04-27 — ISSUE-029 決定に伴いホットリロード/再起動分類セクション追加
**最終更新**: 2026-04-27 — T2 並列呼び出しを単一呼び出し方式に変更（セクション3・4を更新）
**最終更新**: 2026-04-27 — ISSUE-034 決定によりシンボリックリンク記述を削除
```

### 理由

別セッションの AI がドキュメントを読んだ時に「このファイルはいつ・何のために更新されたか」を即座に把握できるようにする。これがないと、自動更新されたことに気づかず古い情報を参照してしまうリスクがある。

### ルール

- **ドキュメント更新時は必ず付記する**（新規作成時を除く）
- 1行で収める（長い説明は不要）
- 複数箇所を変更した場合は主要な変更のみ記載
- `作成日` 行は変更しない（初回作成日を永続的に保持）

## 🤖 サブエージェント実行ルール

**原則**: ドキュメント更新・新規ファイル作成などのファイル書き込みは、**サブエージェント（バックグラウンド実行）に委譲する**。

### 理由

- ファイル書き込みはメインの処理フローと独立している → 並行実行で効率化
- メイン（Claude Code）は判断・対話に集中する
- 完了通知は自動で届くため、待機は不要

### 適用対象

- 決定内容の機能別ドキュメントへの追記
- `Issues.md` の更新（追加・削除）
- `イシュー履歴.md` への履歴追記
- `wiki.md` の更新（リンク集の追加・削除・並び替え）
- `README.md` への Wiki 入口リンク追加
- `CLAUDE.md` への運用ルール追記
- 新規ドキュメントの作成

### 実行方法

```
1. メイン: 「どのファイルにどの内容を書き込むか」を判断
2. メイン: Agent ツール（subagent_type=general-purpose, run_in_background=true）でサブエージェント起動
3. サブエージェント: バックグラウンドで書き込み実行
4. メイン: 完了を待たずに次のステップへ進む
```

### 完了を待たないルール

- バックグラウンド起動後、即座に次のユーザー質問・処理に進む
- 完了通知（task-notification）が届いたら、簡潔に確認するのみ
- 「完了を待ちます」「実行中です」のような待機メッセージは出さない
- 並行で複数のサブエージェントを起動しても OK（独立した書き込みなら並列推奨）

### サブエージェントへの指示テンプレ

ファイル書き込みをサブエージェントに依頼する際、以下を必ず指示する:

```
【タスク】{何のファイルに何を書き込むか}

【書き込み内容】
{具体的な内容、ヒアドキュメント形式で記載}

【守るべきルール】
1. 記載前に Grep で重複チェック（docs/ 配下を全文検索）
2. 重複があれば、マスター判定原則に従い、参照リンク形式で対応
3. マスター更新時は、関連ドキュメントの参照リンクが正しいか確認
4. wiki.md の更新が必要なら同時に対応
5. ファイル末尾の `**最終更新**` 行を `**最終更新**: YYYY-MM-DD — {変更内容の概要}` 形式で更新する
6. 完了したら、何を更新したかを 1〜3 行で報告
```

## 📋 Issue 番号の管理

### 連番ルール

- フォーマット: `ISSUE-001`, `ISSUE-002`, `ISSUE-003`, ...
- **一度発行した番号は使い回さない**（決定済みの番号も履歴で参照可能なため）
- 新規 Issue 追加時は、`Issues.md` と `イシュー履歴.md` の **両方** から最大番号を確認して +1

### 番号取得手順（Grep を使う）

```bash
# Issues.md と イシュー履歴.md から ISSUE-XXX パターンを抽出して最大値を取る
Grep tool: pattern="ISSUE-(\d+)" path="docs/" output_mode="content"
```

抽出した数値の最大値 + 1 が新規 ISSUE 番号。

### サブ Issue

複数のサブ Q を含む大きな Issue は `### ISSUE-XXX-1`, `### ISSUE-XXX-2` のサブ番号で展開する。

## 🔁 検討〜決定〜実装の運用ループ

```
1. [新しい検討事項発生]
   → Issues.md に新規 ISSUE 追加（次の連番）
   → 形式: ## ISSUE-XXX: {タイトル} + 背景・選択肢・推奨方式・理由

2. [ユーザーと打ち合わせ・判断]
   → ユーザーが選択肢を決定

3. [決定内容を機能別ドキュメントに転記]
   → マスター判定原則に従い、適切な機能別ドキュメントを選択
   → 決定経緯（選択肢比較・採用理由）も機能別ドキュメントに記載

4. [Issues.md から削除 + イシュー履歴.md に追記]
   → Issues.md の該当 ISSUE を削除
   → イシュー履歴.md の最下部に追記（決定内容・決定日・転記先）

5. [派生する新規 Issue を追加（あれば）]
   → 決定内容から「さらに詰めるべき細部」が見つかった場合、新規 ISSUE 追加
   → ステップ1に戻る
```

---

## 🚫 やらないこと

- **`docs/` 配下にフォルダを作らない** — すべてのドキュメントは `docs/` 直下に平らに配置する
- **複数のドキュメントに同じ決定内容を書かない** — 1 つの仕様は 1 つのドキュメントで管理する
- **`wiki.md` を更新せずにドキュメントを増やさない** — 新規作成時は必ず `wiki.md` のリンク集も同時に更新
- **古い情報を残さない** — 削除したドキュメントがあれば `wiki.md` のリンクも削除する
- **メインで同期的にファイル書き込みしない** — サブエージェント（バックグラウンド）に委譲するのが原則
