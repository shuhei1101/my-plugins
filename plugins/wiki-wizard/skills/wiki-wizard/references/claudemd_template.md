# CLAUDE.md 統合用テンプレート

このテンプレートは、wiki-wizard を使うプロジェクトの `CLAUDE.md` に追記するための運用ルール集。

**使い方**:
- フェーズ1.5（または独立呼び出し）でユーザーに「CLAUDE.md にこのルールを追記しますか？」と確認
- 既存の `CLAUDE.md` がある場合は、重複セクションを Grep で検出し、追加 / 上書き / スキップを選んでもらう
- ない場合は新規作成

---

## 追記する内容（コピペ用）

以下の Markdown ブロック全体を `CLAUDE.md` に追記する。

````markdown
# CLAUDE.md — ドキュメント管理ガイド

このプロジェクトのドキュメントは [wiki-wizard](https://github.com/) スキルで管理しています。
ドキュメント全体のナビゲーションは [wiki/Home.md](wiki/Home.md) を参照してください。

---

## 📁 ドキュメンツフォルダ構造

### 原則

- **`wiki/` フォルダ配下は平べったく構成**（すべてのドキュメントは同一レベル）
- **ナビゲーションハブは `wiki/Home.md`** で管理（全ドキュメントへのリンク集）
- **プロジェクトルート `README.md`** はプロジェクト概要 + Wiki 入口リンクのみ
- **フォルダを増やさない**（将来的な拡張も `Home.md` のセクション追加で対応）

### 標準ファイル

```
wiki/
├── Home.md            ← ナビゲーションハブ（全ドキュメントへのリンク集）
├── Issues.md          ← 未決定事項・残課題の Issue 集
├── イシュー履歴.md    ← 決定済みイシューの履歴ログ（追記型）
└── {機能別ドキュメント}.md ...
```

---

## 📋 Issue 駆動の意思決定運用

### 役割分担

- `wiki/Issues.md` — **未決定**のイシューのみ（GitHub Issue 的な扱い）
- `wiki/イシュー履歴.md` — **決定済み**のイシューの履歴（追記型ログ）

### Issue 番号の付け方

- **連番管理**: `ISSUE-001`, `ISSUE-002`, `ISSUE-003`, ...
- **一度発行した番号は使い回さない**（履歴で参照可能なため）
- 新規追加時は `wiki/Issues.md` と `wiki/イシュー履歴.md` の両方から最大番号を確認して +1

### 運用フロー

```
1. [新しい検討事項発生]
   → wiki/Issues.md に新規 ISSUE 追加（次の連番）
   → 形式: ## ISSUE-XXX: {タイトル} + 背景 + 選択肢 + 推奨方式

2. [ユーザーと打ち合わせ・判断]
   → 選択肢を決定

3. [決定内容を機能別ドキュメントに転記]
   → 決定経緯（選択肢比較・採用理由）も含めて記載

4. [wiki/Issues.md から削除 + wiki/イシュー履歴.md に追記]
   → wiki/イシュー履歴.md の最下部に「質問内容 / 決定した内容 / 決定日 / 転記先」を追記
```

### wiki/Issues.md の書き方

```markdown
## ISSUE-XXX: {タイトル} {未決定 | 検討中 | 保留}

**背景**: {なぜこのイシューが必要か}

**選択肢**:
- **A**: ...
- **B**: ...
- **C**: ...

**推奨方式**: {A/B/C} — {理由}
```

複数のサブ Q がある場合は `### ISSUE-XXX-1` のサブ番号で展開する。

### wiki/イシュー履歴.md の追記フォーマット

```markdown
## ISSUE-XXX: {タイトル}

- **質問内容**: {元の質問の概要、1〜2 行}
- **決定した内容**: {採用した選択肢と理由の要約、1〜3 行}
- **決定日**: YYYY-MM-DD
- **転記先**: [{ドキュメント名}](wiki/{ファイル名}.md#{セクション名})

---
```

最新の決定が一番下に来る形（追記型・時系列順）。

## 📚 ドキュメント記載時の重複チェックルール

新しい情報を書く前に必ず以下を確認する:

1. **同じ情報が他のドキュメントにすでに書かれていないか？**
   - Grep ツールで該当キーワードを `wiki/` 配下で全文検索
2. **マスタードキュメントの判定原則**:
   - 機能仕様 → 機能別ドキュメント
   - 全モード共通仕様 → 共通仕様.md（プロジェクトに応じて）
   - 未決定事項 → `wiki/Issues.md` がマスター
   - 決定済みの設計判断（経緯含む） → 機能別ドキュメントがマスター
   - API 仕様 → 各 API ドキュメント
3. **マスター以外への記載は「参照リンク」に統一**

詳細は wiki-wizard の `references/workflow_rules.md` を参照。

## 🤖 ファイル書き込み実行ルール

**原則**: ドキュメント更新・新規ファイル作成などのファイル書き込みは、サブエージェント（バックグラウンド実行）に委譲する。

- メインは判断・対話に集中
- バックグラウンド起動後、完了を待たずに次のステップへ進む
- 完了通知は自動で届くため、待機メッセージは不要

---

## 🚫 やらないこと

- `wiki/` 配下にフォルダを作らない
- 複数のドキュメントに同じ決定内容を書かない
- `Home.md` を更新せずにドキュメントを増やさない
- 古い情報を残さない（削除時は `Home.md` のリンクも削除）
````

## 追記時の注意

- 既存の `CLAUDE.md` に同名セクション（例: 「📁 ドキュメンツフォルダ構造」）がある場合は、ユーザーに確認してから上書き / マージ
- 既存の `CLAUDE.md` がない場合は、上記をそのまま `CLAUDE.md` として作成して構わない
- プロジェクト固有のセクション（例: 「Python 実装ルール」など）と混在する場合は、上記ブロックを最上部または専用セクションとして追記し、既存内容は保持する
