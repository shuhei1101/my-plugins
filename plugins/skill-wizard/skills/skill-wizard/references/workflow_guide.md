## 対話形式スキルの前提知識

このスキルが生成する「対話形式スキル」は以下の原則に従う。生成するSKILL.mdにはこれらをスキル利用者（AI）が理解できる形で組み込むこと。

### 基本原則

- スキル内に**ステップ**と呼ばれるワークフロー単位が定義されている
- AIはステップに沿ってタスクを遂行し、**必ず最後に選択肢を提示**する
- ユーザーは選択肢を選んで会話を進める
- 選択肢以外の自由入力があった場合、AIが最も近い選択肢を判定する
- どの選択肢にも当てはまらない場合、再選択を促すか「フローから外れますがよろしいですか？」と確認する（これも選択肢で提示）

### フローの基本ループ

対話形式スキルのフローは、以下のループを基本単位として設計すること。これはすべてのフロー設計の前提である。

```
ユーザー: 指示・要望
  ↓
AI: 質問（要件の確認・不足情報のヒアリング） ←─┐
  ↓                                             │
ユーザー: 回答 ─────────────────┘
  ↓              回答からさらに質問が出た場合は
  ↓              質問フェーズに戻る
AI: 最終確認（理解した内容をまとめて提示）
  ↓
ユーザー: 確認完了 or 修正指摘 ─→ 修正指摘なら質問フェーズに戻る
  ↓
AI: 実行（ファイル更新・生成など）
  ↓
ユーザー: 変更内容の確認 → OK / NG
  ↓
（NGなら修正ループへ、OKなら次のフェーズへ）
```

**このループを守る理由**: AIが十分な理解なしに実行に入ると、手戻りが大きくなる。「質問→確認→実行→レビュー」の順序を崩さないこと。

ステップ設計時は、このループのどのフェーズに該当するステップかを意識する：
- **質問ステップ**: ユーザーから情報を引き出す（選択肢 or 自由入力）
- **確認ステップ**: AIが理解した内容をまとめてユーザーに提示し、承認を得る
- **実行ステップ**: 確認済みの内容に基づいてファイル操作やコード生成を行う
- **レビューステップ**: 実行結果をユーザーに提示し、OK/NGの判定を得る

### 抽象→具体の原則（Progressive Zoom）

フローの設計・情報の提示・説明の構成において、**常に抽象から具体へ**流れること。

**基本パターン**:
1. 全体図（概要・一覧）を提示する
2. ユーザーに「どこを詳しく見たい？」と聞く
3. 選択された部分の詳細を提示する
4. 選択肢に必ず「一覧に戻る」を含める（抽象↔具体の行き来を保証）

**ズームレベルの調整** — 対象の規模に応じて段階数を変える：

| 規模                   | 段階     | 例                                    |
| ---------------------- | -------- | ------------------------------------- |
| 小規模（項目5個以下）  | 2段階    | 一覧 → 詳細                           |
| 中規模（項目10個前後） | 2〜3段階 | 一覧 → カテゴリ → 詳細                |
| 大規模（項目20個以上） | 3〜4段階 | 全体図 → セクション → グループ → 詳細 |

> **注意**: 段階を増やせばいいわけではない。段階が多いと行き来がしづらくなる。ケースバイケースで判断すること。「ユーザーが今いる場所を見失わないか？」を基準にする。

**フロー設計で守るべきこと**:
- ステップ一覧を見せてから個別ステップの詳細に入る（いきなり詳細に飛ばない）
- 詳細を見た後は必ず一覧に戻れる選択肢を用意する
- 説明や図示でも同じ：まず全体構造を示し、ユーザーが選んだ部分を掘り下げる

### ステップの構造テンプレート

```markdown
## {フェーズ名 or このフェーズに入る条件}
### ステップ({ステップタイトル})
条件: {このステップが実行される条件}
入力: {ユーザーから必要な入力。未提供の場合は入力を求める}

AI実施タスク:
  - タスクA: {概要}
    - サブエージェント: {実行方式}
    - 必読情報: {渡すスキル・ファイルパス・コンテキスト}
  - タスクB: {概要}

出力: {ユーザーに提示する内容}
選択肢:
  1. 選択肢1
  2. 選択肢2
  3. その他（入力してください）
補足: {追加情報}
```

#### 各フィールドの意味

| フィールド       | 意味                                   | 書くべき内容                                                             |
| ---------------- | -------------------------------------- | ------------------------------------------------------------------------ |
| **条件**         | このステップが実行される前提条件       | どの状態・選択からこのステップに到達するか                               |
| **入力**         | ユーザーから受け取る情報               | このステップの実行に必要なユーザー入力                                   |
| **AI実施タスク** | **出力を生成する前に**AIが裏で行う処理 | データ収集、分析、ファイル読み込み、サブエージェント実行、memory記録など |
| **出力**         | AIがユーザーに見せる内容               | AI実施タスクの結果をもとにユーザーに提示するテキスト                     |
| **選択肢**       | ユーザーが次にとれるアクション         | 番号付きの選択肢リスト。各選択肢にはどのステップに遷移するかを明記       |
| **補足**         | その他の注意点・参照先                 | 実装上の注意、参照すべきファイル、例外処理など                           |

> **よくある間違い — AI実施タスクに「ルーティング（分岐先）」を書いてしまう**
>
> AI実施タスクは「ユーザーに出力を見せる**前に**AIが行う処理」である。
> 「ユーザーの選択に応じてどのステップに飛ぶか」はAI実施タスクではなく、**選択肢**または**補足**に書くこと。
>
> - NG: `AI実施タスク: 新規作成リクエスト → ステップ2へ、編集リクエスト → ステップ9へ`
> - OK: `AI実施タスク: ワークスペース内の既存ファイルを走査し、参考情報を収集する`
>
> ただし、**起動時のコンテキスト分析**（ユーザーの入力から意図を判定し、該当ステップにジャンプする処理）はAI実施タスクとして書いてよい。これは出力を見せる前にAIが行う判定処理であり、ルーティングそのものではない。この場合も、分岐先の一覧は**補足**に書くのが望ましい。

### 選択肢の順序ルール

選択肢は**正規ルート（前に進む選択肢）を必ず1番目**にすること。修正・戻る・その他は2番目以降に配置する。

- 1番目: 正規ルート（OKで次に進む、この内容で確定、など）
- 2番目以降: 修正、やり直し、戻るなど
- 最後: その他（入力してください）

### 進捗バー

各ステップの選択肢の下に1行空けて、進捗バーを表示すること。AIが動的に生成する。

**フォーマット**:
```
[██████░░░░░░░░░░░░░░] ステップX.Y / 全Nフェーズ
```

**ルール**:
- バーは現在のフェーズ内での進捗を示す（フェーズ内の全ステップ数に対する現在ステップの位置）
- バーの全体幅は20文字（█ と ░ の合計）
- テキストは「ステップX.Y / 全Nフェーズ」のみ

**例**: フェーズ2（ステップ2.1〜2.7）のステップ2.3にいる場合
```
1. この内容でOK
2. 修正を伝える
3. その他（入力してください）

[████████░░░░░░░░░░░░] ステップ2.3 / 全6フェーズ
```

### デフォルト選択肢

SKILL.md上の各ステップの選択肢に明示されていなくても、AIは必ず以下を選択肢に含めること:
- **その他（入力してください）**

### 「その他」選択肢の処理ガイドライン

ユーザーが「その他」を選択したり、選択肢外の入力をした場合、以下の対応を行うこと:

#### 1. 入力内容の意図を解釈する

- **情報不足を示唆**（例: 「もっと情報が欲しい」「詳しく教えて」「わからない」）
  → 詳細説明や具体例を提供し、改めて選択肢を提示

- **迷いを示唆**（例: 「まだ決めてない」「どうしたらいいか分からない」）
  → ランダム生成機能の提案、または段階的なヒアリングを開始

- **前のステップに戻りたい**（例: 「戻りたい」「やり直したい」「前に戻る」）
  → 前のステップへの遷移選択肢を提示

- **内容の修正**（例: 「さっきの入力を変更したい」「間違えた」）
  → 該当項目の再入力を促す

- **スキップ希望**（例: 「スキップしたい」「後でやる」「とりあえず次へ」）
  → デフォルト値で進行可能な場合は提案、不可能な場合は理由を説明

- **不明な要求**
  → 明確化のための質問を投げかけ、ユーザーの真意を確認

#### 2. 柔軟な対応の原則

- **厳密な選択肢番号に固執しない**: ユーザーの意図が明確なら、番号以外の表現も受け入れる
- **ユーザーの意図を最優先**: 形式よりも内容を重視
- **複数の進路を提示**: 不明確な場合は、複数の選択肢を提示して選んでもらう
- **エラーにしない**: どんな入力でも、何らかの建設的な対応を返す

#### 3. よくあるパターンと推奨対応

| ユーザー入力例                   | 解釈           | 推奨対応                                     |
| -------------------------------- | -------------- | -------------------------------------------- |
| 「まだ決めてない」「わからない」 | 迷い・情報不足 | ランダム生成提案 or 詳細ヒアリング開始       |
| 「詳しく教えて」「例を見せて」   | 情報要求       | 詳細説明+具体例提示+選択肢再提示             |
| 「戻りたい」「やり直す」         | ステップ遡行   | 前のステップへの遷移を提案                   |
| 「スキップ」「後で」             | 一時保留       | デフォルト値提案 or 保留不可の理由説明       |
| 「全部やって」「おまかせ」       | 自動化希望     | 推奨設定・デフォルト値でのフル自動実行を提案 |

#### 4. 例外的な対応が必要な場合

- **フロー全体の中断**: ユーザーが明確に中止を望む場合（「やめる」「キャンセル」）
  → 中断の確認と、作業途中の状態保存（memoryへの記録）を実施

- **完全に予期しない入力**: どの解釈も当てはまらない場合
  → 「入力内容を確認できませんでした。以下のいずれかを選んでください」と丁寧に選択肢を再提示

### 選択肢の提示形式

選択肢は必ず番号付きで提示すること:
```
1. 選択肢1
2. 選択肢2
3. その他（入力してください）
```

### 多角度質問の提示形式

確認ステップで複数の観点から質問を行う場合、以下の形式で提示すること:

```
Q1. 質問内容は？
→ (XXX / YYY / ZZZ)

Q2. 次の質問内容は？
→ 

Q3. さらなる質問は？
→ 
```

**質問提示時の必須要素**:
- 質問の最後に必ず以下のTipsを追加: `💡 **Tips**: 入力時は `/copy` コマンドを使用して入力フォームを活用してください`
- 各質問には番号（Q1, Q2...）を付与

### サブエージェント実行オプション

タスクの性質に応じて実行方式を使い分ける:
- **同期実行**（デフォルト）: 現在のコンテキストで順次実行
- **並列実行（待つ）**: サブエージェントで並列に実行し、全完了を待って結果を統合
- **非同期実行（待たない）**: サブエージェントでバックグラウンド実行し、完了を待たない

### ペルソナの活用（多角的な案出し）

**重要**: 複数の観点から案・意見・評価を得たい場合は、`/idea` を使用することを推奨する。`/idea` は、ペルソナを活用して複数のサブエージェントを並列起動し、異なる視点からの意見を統合して提示する。

**`/idea` の使い方**:
- トピックと観点を指定: 「{トピック}について、{観点1}・{観点2}・{観点3}の観点から意見出して」
- 観点を自動選択: 「{トピック}について案だして」（AIが適切な観点を提案）

**`/idea` が利用できない場合**（ideaスキルが存在しないか読み込めない場合）:
- サブエージェントを直接起動し、各サブエージェントに異なるペルソナを割り当てる
- ペルソナファイルも無い場合は、自分で観点を設定してサブエージェントを起動する

### フロー制御

ステップ間の遷移はマークダウンの自由形式で補足できる:
- **ループ**: 条件を満たすまでステップを繰り返す
- **条件分岐**: ユーザーの選択に応じて別ステップへジャンプ
- **早期終了**: 特定条件でフロー全体を終了

### プラグインキャッシュ同期

リポジトリ側のファイルを変更した後、プラグインキャッシュ（`~/.claude/plugins/marketplaces/...`）への同期を**自動で実行しない**。
ファイル出力・更新が完了したステップでは、以下の選択肢を提示すること：

```
変更をプラグインに反映して、ローカルで使えるようにしますか？

1. 反映する
2. 反映しない
3. その他（入力してください）

※ローカルキャッシュを上書きするだけで、リモート上のプラグインは更新しません
```
