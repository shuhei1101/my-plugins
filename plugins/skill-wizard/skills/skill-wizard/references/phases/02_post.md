# 全ステップ確認完了後フェーズ

すべてのステップの詳細設計が完了した後、関連ファイルの整理、構造チェック、最終出力を行います。

---

## ステップ3.1(関連ファイルの確認)

**条件**: 全ステップの詳細確認が完了

**入力**: -

**AI実施タスク**:
  - 設計済みステップの内容から、参照すべき関連ファイルのパスと概要を提案する
  - **重要**: ファイルの実際の内容は含めず、パスと概要のみを提案する。これにより、SKILL.mdを軽量に保ち、必要な時だけファイルを読み込んでコンテキスト超過を防ぐ

**出力**: 関連ファイルの提案を表形式で提示

  | ファイル名   | 概要                    | パス              |
  | ------------ | ----------------------- | ----------------- |
  | xxx_guide.md | XXXの実行ガイド         | docs/xxx_guide.md |
  | helper.py    | XXX処理の支援スクリプト | src/helper.py     |
  
  これらのパスをSKILL.mdの「関連ファイル」セクションに記載します。

**選択肢**:
  1. これでOK
  2. 修正を伝える（追加・削除・変更）
  3. その他（入力してください）

**補足**: 
  - SKILL.md内には「詳細は `docs/xxx_guide.md` を参照」のように記載し、実際の内容は必要時に読み込む方式でコンテキストを節約する
  - **ファイル配置の判断基準は `references/file_organization.md` を参照すること**:
    - `scripts/`: 実行可能なコード（決定的・反復的なタスク用）
    - `references/`: 必要に応じてコンテキストに読み込むドキュメント
    - `assets/`: 出力に使用するファイル（テンプレート、アイコンなど）

## ステップ3.2(構造チェックと最適化)

**条件**: 関連ファイルの確認完了後、出力前

**入力**: -

**AI実施タスク**:
  - SKILL.mdの行数をカウントする
  - 重複コンテンツの検出:
    - 同じ内容が複数箇所に記載されていないかチェック
    - 似た説明や冗長な記述を洗い出す
    - 重複箇所を特定し、統合または削除を提案
  - 500行を超える場合（空白行を含む。`wc -l`や`Measure-Object -Line`でカウント）:
    - **優先: Index-style構造（フェーズ分割）を提案**
      - 対話形式スキルでフェーズが明確な場合、skill-wizard自身のような構造を推奨
      - SKILL.mdは150行以下のindexとし、各フェーズを `references/phases/XX_{フェーズ名}.md` に分割
      - 各フェーズファイルには完全なステップ定義（条件・入力・AI実施タスク・出力・選択肢・補足）を含める
      - SKILL.md内の各フェーズには `📁 **必読**: references/phases/XX_{フェーズ名}.md` を明記
    - **代替: セクション単位の分離**
      - 単発スキルやフェーズ分割に適さない場合
      - 長大なセクションを特定する（例: 詳細な手順、サンプルコード、補足説明、テンプレート等）
      - references/フォルダへ分離可能なセクションを抽出する
      - 各セクションの行数と概要を表形式でまとめる
  - descriptionの形式チェック:
    - トリガーフレーズが複数列挙されているか
    - 「〜という依頼で起動」または「〜といったリクエストで起動」で終わっているか

**出力**:
  - 行数報告（例: 「SKILL.mdは現在523行です」）
  - 重複コンテンツがある場合、重複箇所の表を提示:

  | 番号 | 重複内容                 | 出現箇所                   | 推奨アクション         |
  | ---- | ------------------------ | -------------------------- | ---------------------- |
  | 1    | ステップ構造の説明       | 「前提知識」と「詳細設計」 | どちらかに統合して削除 |
  | 2    | サブエージェント実行方式 | 複数ステップの補足         | 「前提知識」に集約     |
  | ...  | ...                      | ...                        | ...                    |

  - 500行超過の場合、構造パターンを提案:

  **推奨: Index-style構造（対話形式スキルの場合）**
  
  フェーズ構成案:
  | フェーズ  | 内容         | 行数   | 移動先ファイル                  |
  | --------- | ------------ | ------ | ------------------------------- |
  | フェーズ1 | {フェーズ名} | {行数} | references/phases/00_{名前}.md |
  | フェーズ2 | {フェーズ名} | {行数} | references/phases/01_{名前}.md |
  | ...       | ...          | ...    | ...                             |
  
  SKILL.md本体は150行以下のindexとなり、各フェーズに `📁 **必読**: references/phases/XX.md` を記載します。

  **代替: セクション分離（単発スキルまたはフェーズ分割不適切な場合）**
  
  | 番号 | セクション名             | 行数 | 概要                         | 移動先候補                |
  | ---- | ------------------------ | ---- | ---------------------------- | ------------------------- |
  | 1    | ステップ構造テンプレート | 25   | マークダウンのテンプレート例 | references/template.md    |
  | 2    | セッション履歴サンプル   | 18   | セッション履歴の記載例       | references/session.md     |
  | 3    | テスト結果ファイル形式   | 30   | テスト出力のマークダウン例   | references/test_format.md |
  | ...  | ...                      | ...  | ...                          | ...                       |

  - descriptionの形式に問題がある場合、修正案を提示

**選択肢**:
  1. このままでOK（500行以下、または分離不要）
  2. 重複を削除する（番号で指定、複数可）
  3. Index-style構造（フェーズ分割）を適用する → フェーズごとに分割ファイルを作成
  4. セクション分離を適用する（番号で指定、複数可）
  5. descriptionを修正する
  6. その他（入力してください）

**補足**:
  - 重複削除が選択された場合、該当箇所を統合または削除し、SKILL.mdを更新する
  - **Index-style構造が選択された場合**:
    - 各フェーズを `references/phases/XX_{フェーズ名}.md` ファイルに分割
    - 各ファイルには完全なステップ定義（条件・入力・AI実施タスク・出力・選択肢・補足）を記載
    - SKILL.md本体は150行以下のindexに圧縮し、各フェーズに `📁 **必読**: references/phases/XX.md` を明記
    - **条件フィールドは各ステップに保持**（フェーズヘッダーに統合しない）
  - セクション分離が選択された場合、該当セクションをreferences/配下のファイルに移動し、SKILL.md内には「詳細は `references/xxx.md` を参照」と記載する
  - 変更後は再度行数をカウントして報告する
  - **推奨: 対話形式スキルはIndex-style構造（150行以下）、単発スキルは単一ファイル（500行以下）**

## ステップ3.5(プラグインJSON更新)

**条件**: SKILL.md出力完了後

**入力**: スキルの出力先パス、スキル名、description

**AI実施タスク**:
  - スキルの出力先から親プラグインディレクトリを特定する
  - **plugin.json の確認・作成**:
    - `{プラグインディレクトリ}/.claude-plugin/plugin.json` が存在するか確認
    - 存在しない場合、以下の内容で新規作成:
      ```json
      {
        "name": "{プラグイン名}",
        "description": "{プラグインの説明}",
        "version": "1.0.0",
        "author": "{ユーザー名またはデフォルト値}"
      }
      ```
    - 既に存在する場合、description や version を必要に応じて更新
  - **marketplace.json の確認・更新**:
    - 親マーケットプレースの `marketplace.json` が存在するか確認（通常 `{マーケットプレースルート}/marketplace.json`）
    - 存在する場合、plugins 配列に新しいプラグインエントリを追加:
      ```json
      {
        "name": "{プラグイン名}",
        "path": "{プラグインの相対パス}",
        "description": "{プラグインの説明}"
      }
      ```
    - 同名エントリが既に存在する場合は、description を更新
    - marketplace.json が存在しない場合はスキップし、ユーザーに通知
  - 作成・更新した内容をユーザーに報告

**出力**: JSON更新結果の報告

  ```
  ✅ plugin.json: {作成 / 更新 / 変更なし}
     パス: {plugin.jsonのパス}
  
  ✅ marketplace.json: {エントリ追加 / エントリ更新 / スキップ（ファイルなし）}
     パス: {marketplace.jsonのパス}
  ```

**選択肢**:
  1. テストを実行する → ステップ5.1（テストフェーズ）へ
  2. 完了 → プラグイン反映の確認へ
  3. JSON内容を確認する
  4. 修正する（フィールド値を指定）
  5. その他（入力してください）

**補足**:
  - plugin.json の author フィールドは、git config の user.name や環境変数から推定を試みる
  - 既存スキルの編集時は version を必ずインクリメントする（パッチ: 軽微な修正 例 1.0.0→1.0.1、マイナー: 機能追加・ステップ変更 例 1.0.0→1.1.0）。marketplace.json側も同じ値に更新する
  - marketplace.json への重複登録を防ぐため、name フィールドで既存エントリを検索してから追加する
