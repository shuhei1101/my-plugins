# メインフェーズ

このフェーズでは、スキルの詳細設計を行います。スキル名の決定、ステップ構成の設計、各ステップの詳細を詰めていきます。

---

## ステップ2.1(スキル概要の理解とスキル名提案)

**条件**: -

**入力**: スキル概要（ユーザーが伝える。未入力の場合は入力を促す）

**AI実施タスク**:
  - スキル概要の理解と整理
  - 不明点・確認事項の洗い出し
    - **`/idea` を使用**して、複数の観点から質問を生成
      - トピック: 「{スキル概要}の不明点・確認事項」
      - 観点: ユーザー体験・技術的実現性・エッジケース等（スキルの性質に応じて自動選択）
    - **`/idea` が利用できない場合**:
      - サブエージェント: 2〜3体を並列実行
      - ペルソナが無い場合: 自分で観点を設定（例: 「ユーザー体験」「技術的実現性」「エッジケース」）
  - スキル名を提案する
    - 複数の命名視点（キャッチー重視・シンプル重視・機能明瞭・1単語重視など）から名前案を生成
    - 各視点から3〜5つずつ名前案を生成（ケバブケース、1つは1単語）
    - 重複を除いて10個程度に絞る
    - 推奨候補には ⭐ [推奨] マークをつける
  - `~/.claude/skill-memory/skill-wizard/history/{YYYYMMDDHHMMSS}_skill_creation.md` を更新:
    ```
    ### スキル名候補
    1. {候補1} ⭐
    2. {候補2}
    ...
    
    ### 質問事項
    - {質問1}
    - {質問2}
    ```

**出力**:
  - スキル概要のまとめ
  - スキル名候補を表形式で提示:

  | No. | 名前         | 推奨 | 視点     | 理由                                    |
  | --- | ------------ | ---- | -------- | --------------------------------------- |
  | 1   | wizard       | ⭐   | 工数最小 | 1単語で最短。対話的に進める定番パターン |
  | 2   | step-builder |      | 実装者   | 機能が明確で検索しやすい                |
  | ... | ...          | ...  | ...      | ...                                     |

**選択肢**:
  1. スキル名を選択してメニュー構成案へ進む（番号で指定）→ ステップ2.1bへ
  2. 次のステップに進む（質問ステップ）→ ステップ2.2へ
  3. 概要を補足・修正する
  4. もっと違う視点から名前案が欲しい
  5. その他（入力してください）

**補足**:
  - ユーザーがすでにスキル名を決めている場合は、名前提案をスキップしてステップ2.1bへ進む
  - ユーザーの説明は往々にして「適当」。いろんな角度から質問して、本当に何を望んでいるかを掘り下げる。質問しているうちにユーザー自身がひらめくことも多い
  - **質問は必ず「多角度質問の提示形式」に従う**こと。各質問に適切な矢印を選択し、最後に `/copy` のTipsを必ず添える
  - **質問と選択肢は同じステップに同居させない**こと。質問が必要な場合は次の質問ステップへ遷移させる
  - 名前案生成は複数の視点を組み合わせることで、ユーザーが思いつかなかった選択肢を提示できる

## ステップ2.1b(メニュー構成案の提案)

**条件**: スキル名が確定済み（ステップ2.1でスキル名を選択後、またはユーザーが事前にスキル名を決めていた場合）

**入力**: スキル概要、確定したスキル名、これまでの質問回答（あれば）

**AI実施タスク**:
  - スキル概要と質問回答をもとに、メインメニュー（ステップ1.1相当）の選択肢一覧を提案する
  - 各選択肢について、機能の概要を1行で説明する
  - **`/idea` を使用**して、ユーザーが思いつかなかった追加機能を提案する
    - トピック: 「{スキル名}のメニュー構成・機能候補」
    - 観点: ユーザー体験重視・拡張性重視・実用性重視・差別化重視等（スキルの性質に応じて自動選択）
    - `/idea` の結果から有用な機能案を抽出し、「AIおすすめ」として提示する
  - **`/idea` が利用できない場合**:
    - サブエージェント: 2〜3体を並列実行
    - 各サブエージェントに異なる観点（例: 「パワーユーザー向け機能」「初心者向け機能」「運用・保守向け機能」）を割り当てる
    - 結果を統合して追加機能候補を生成する
  - `~/.claude/skill-memory/skill-wizard/history/{YYYYMMDDHHMMSS}_skill_creation.md` を更新:
    ```
    ### メニュー構成案
    1. {機能A} — {機能Aの説明}
    2. {機能B} — {機能Bの説明}
    ...
    
    ### AIおすすめ追加機能
    - {追加機能1} — {理由}
    - {追加機能2} — {理由}
    ```

**出力**:
  - メインメニューの選択肢一覧を表形式で提示:

  | No. | 選択肢                         | 機能概要                       | 備考           |
  | --- | ------------------------------ | ------------------------------ | -------------- |
  | 1   | {機能A}                        | {機能Aの説明}                  |                |
  | 2   | {機能B}                        | {機能Bの説明}                  |                |
  | ... | ...                            | ...                            |                |
  | N   | その他（入力してください）     | 自由入力                       |                |

  - AIおすすめ追加機能（ユーザーが明示していないが有用と判断した機能）:

  | No. | 追加機能案                     | 概要                           | おすすめ理由                   |
  | --- | ------------------------------ | ------------------------------ | ------------------------------ |
  | A1  | {追加機能1}                    | {説明}                         | {なぜあると便利か}             |
  | A2  | {追加機能2}                    | {説明}                         | {なぜあると便利か}             |
  | ... | ...                            | ...                            | ...                            |

**選択肢**:
  1. このメニュー構成でOK → 質問ステップ（ステップ2.2）へ
  2. おすすめ追加機能を採用する（番号で指定、複数可）→ メニューに追加して再提示
  3. 機能の追加・削除・変更を伝える → 修正して再提示
  4. スキル概要に立ち返って見直す → ステップ2.1へ
  5. その他（入力してください）

**補足**:
  - この一覧がスキルのメインメニュー（ステップ1.1）の選択肢のベースになる
  - 「その他（入力してください）」は必ず末尾に含める
  - AIおすすめ追加機能では、ユーザーの要望にはなかったが「あったら便利」な機能を積極的に提案する。ただし過剰にならないよう3〜5個程度に絞る
  - ユーザーが追加機能を採用した場合は、メニュー一覧に統合して再提示する

## ステップ2.2(質問ステップ)

**条件**: ステップ2.1またはステップ2.1bから「質問ステップ」が選択された場合、またはAIが質問への回答を先に得る必要があると判断した場合

**入力**: ステップ2.1で整理した不明点・確認事項

**AI実施タスク**:
  - 複数の観点から質問を整理する
  - 「多角度質問の提示形式」に従って質問文を組み立てる
  - 回答しやすいように、例や補足が必要な質問には短い例を添える
  - 回答後に各質問の回答有無を確認し、回答漏れがあるか判定する

**出力**:
  - 複数の観点からの質問（以下の形式で提示）:
    ```
    Q1. ターゲットユーザーは誰？
    → 

    Q2. 想定する具体的なユースケースは？
    → （例: 「データ処理用のツール」「社内申請ツール」など）

    Q3. MVP範囲の確認
    → 

    💡 **Tips**: 質問をまとめてコピーするには `/copy` コマンドを使ってください
    ```

**選択肢**: なし（このステップでは自由入力で回答してもらう）

**補足**:
  - **このステップでは選択肢を出さない**。質問への回答に集中してもらうため
  - 質問と選択肢は同一ステップに混在させない
  - 回答漏れがある場合はステップ2.3（回答漏れの再確認）へ進む
  - 全質問に回答がそろったらステップ2.4へ進む

---

## ステップ2.3(回答漏れの再確認)

**条件**: ステップ2.2の回答を確認した結果、未回答または回答が不十分な質問がある場合

**入力**: ステップ2.2で未回答または情報不足と判定された質問一覧

**AI実施タスク**:
  - 未回答・情報不足の項目だけを抜き出して再提示する
  - どの質問が未回答なのか一目でわかるように整理する
  - 必要に応じて、回答例や補足を短く添えて再入力しやすくする

**出力**:
  - 再確認が必要な質問一覧
  - 「未回答の項目だけ回答してください」と明示する

**選択肢**: なし（このステップでは自由入力で回答してもらう）

**補足**:
  - **このステップでも選択肢は出さない**
  - 回答漏れが解消しない限り、このステップを再実行してよい
  - 回答がそろったらステップ2.4へ進む

## ステップ2.5(ステップ一覧の提案)

**条件**: スキル概要とスキル名が確定済み

**入力**: スキル概要、スキル名

**AI実施タスク**: 
  - スキル概要から最適なステップ構成を提案する。以下の手順で検討:
    1. **全体のステップの流れを設計する**
       - 先頭に**メインステップ**（モード選択）を設ける。ここでこのスキルのどのモードで進めるか選択させる（例: メイン処理、前回セッション再開など）。ただし、ユーザーが既にやりたいことを明示している場合はスキップ可能とする条件も記載すること
    2. **ファイル編集ステップの前に「確認ステップ」を挿入する**
       - 確認ステップでは変更内容を表形式等でユーザーに提示し、承認を得てから実行に進む
    3. **末尾に「ドキュメント出力ステップ」を追加する**
       - 今回の実行内容をまとめてドキュメント化するステップ
       - 出力先のファイルパスは、このステップの前に確認ステップを入れて決定する
    4. **memory活用を組み込む**
       - 案検討や機能設計などの重要な段階では、`~/.claude/skill-memory/{スキル名}/history/{YYYYMMDDHHMMSS}_{作業内容}.md` に情報を保存する
       - 例: `~/.claude/skill-memory/my-skill/history/20260402143530_feature_design.md`
  - `~/.claude/skill-memory/skill-wizard/history/{YYYYMMDDHHMMSS}_skill_creation.md` を更新:
    ```
    ### ステップ構成案
    1. {ステップ1.1概要}
    2. {ステップ1.7概要}
    ...
    ```

**出力**: ステップ一覧（番号付き、各ステップの概要を1行で）。フェーズごとにグルーピングして表示する:
  ```
  **フェーズ1: 事前準備**
  - ステップ1.1: モード選択（メイン処理 / 前回セッション再開）
  - ステップ1.7: ○○の確認

  **フェーズ2: メイン処理**
  - ステップ2.1: ○○の設計
  - ステップ2.4: ○○の確認
  ...

  **フェーズ3: 出力**
  - ステップN: ドキュメント出力
  ```

**選択肢**:
  1. この一覧でOK → 各ステップの詳細確認へ
  2. ステップの修正を伝える（例: 「ステップ2.1は不要」「ステップ2.5の後にXXXを追加」「○○のタスクも入れたい」）
  3. スキル概要の更新や追加情報を伝える → このステップを再実行
  4. その他（入力してください）

## ステップ2.6(ステップ内容の詳細設計)

**条件**: 全ステップの確認が完了するまでループ

**入力**: -

**AI実施タスク**:
  - 対象ステップの詳細を起案する（条件・入力・AIタスク・出力・選択肢・補足のすべて）
  - タスク設計の観点:
    - **ファイル編集**があるタスクには、関連ドキュメントの更新タスクもセットで入れる。ドキュメント更新と本体編集は多くの場合サブエージェントで並列実行できる
    - **設計・概要理解**のステップでは、`/idea` を使って複数の観点から質問を生成させる
    - **サブエージェント起動時**には、渡す情報（スキル・ファイルパス・コンテキスト）を明記する。AIが現在使えるスキルからおすすめを選択するとともに、ユーザーからも追加指定を受け付ける
    - **多角的な案出しが必要な場合**: `/idea` を使用することを推奨
      - 実装・技術選定 → 「{トピック}について、コスト重視・品質重視・セキュリティ重視の観点から意見出して」
      - ドキュメント作成 → 「{トピック}について、作成者目線・初見読者目線・保守者目線の観点から意見出して」
      - プロジェクト管理 → 「{トピック}について、PL目線・実装者目線・新規参画者目線の観点から意見出して」
    - **memory活用**: 案検討・機能設計など重要な決定を行うステップでは、その内容を `~/.claude/skill-memory/{スキル名}/history/{YYYYMMDDHHMMSS}_{内容}.md` に保存するタスクを含める
  - コンテキスト管理:
    - 長大なコンテキストは外部ファイルにmemoryとして書き出すことを検討する
    - サブエージェント活用でメインコンテキストの圧迫を防ぐ
  - `~/.claude/skill-memory/skill-wizard/history/{YYYYMMDDHHMMSS}_skill_creation.md` を更新:
    ```
    ### 詳細設計済みステップ
    - [✓] ステップ1.1: {概要}
    - [✓] ステップ1.7: {概要}
    - [ ] ステップ2.1: {概要}
    ```

**出力**: ステップ詳細を「ステップの構造テンプレート」（前提知識セクション参照）の形式で表示

**選択肢**:
  1. この内容でOK → 次のステップの設計へ
  2. 修正内容を伝える → 修正して再提示
  3. ステップ一覧を確認する → ステップ一覧表示へ
  4. このステップのカスタムサブエージェントを設定する → ステップ2.6bへ
  5. その他（入力してください）

**補足**:
  - ユーザーの発言は「いい意味で疑う」。表面的な言葉だけでなく、色々な角度から提案して本当に望んでいることを引き出す
  - 確認フェーズと案出しフェーズは手厚く設ける。確認ステップでは選択肢を充実させ、ユーザーが細かく方向性を指示できるようにする
    - **確認ステップでの選択肢生成**: `/idea` を使って「こんなのはどうですか？」というおすすめの選択肢を複数の観点から生成させる（ユーザー体験重視、実装効率重視、柔軟性重視等）。ユーザーが思いつかなかった選択肢を提示する
    - **質問の形式**: 複数の観点から質問を行う際は、必ず「多角度質問の提示形式」に従う。各質問に適切な矢印を選択し、最後に `/copy` のTipsを必ず添える
    - **質問ステップの分離**: 質問を出すステップでは選択肢を併記しない。両方が必要な場合は前後のステップに分け、選択肢側では `1. 次のステップに進む（質問ステップ）` のように案内する
    - **回答漏れの再確認**: 質問ステップの後には、未回答・情報不足の項目だけを確認する再確認フェーズを必要に応じて挟めるようにする
  - **サブエージェントを使うタスクの設計時**: AIタスクにサブエージェントが含まれる場合、選択肢4「カスタムサブエージェントを設定する」を必ず提示すること
  - 2ステップおきに「このタスクで使うコンテキスト・スキルの確認」を挟むことを検討する。ただしメインコンテキスト圧迫を避けるため、主にサブエージェント起動時のコンテキスト指定として組み込む

## ステップ2.6b(カスタムサブエージェント管理 — 詳細設計中)

**条件**: ステップ2.6で「このステップのカスタムサブエージェントを設定する」が選択された場合

**入力**: 現在設計中のステップ情報（ステップ名・AIタスク概要）

**AI実施タスク**:
  - 現在設計中のステップでのサブエージェントの役割を確認する
  - 対象スキルの `agents/` フォルダに既存エージェントがあるか確認する
    - 既存エージェントがある場合: 一覧を表示し、活用可能かを提案する
    - 既存エージェントがない場合: 新規作成を提案する
  - `/idea` のペルソナが参考になるか判断する（実装/ドキュメント/プロジェクト/レビュー/命名のいずれか）
  - `references/custom_agents_guide.md` を参照して、適切なエージェント設計パターンを提案する

**出力**:
  - 現在のステップに必要なサブエージェントの役割まとめ
  - 既存エージェントの活用案（あれば）または新規作成案

**選択肢**:
  1. 既存エージェントをこのステップに紐づける（エージェント名を指定）→ ステップ6.3へ
  2. 新しいエージェントを作成してこのステップに紐づける → ステップ6.1へ（対象タスクを引き継ぐ）
  3. `/idea` のペルソナをそのまま使う（agents/フォルダは不要）→ ステップ2.6に戻る
  4. このステップはサブエージェント不要と判断した → ステップ2.6に戻る
  5. その他（入力してください）

**補足**:
  - ステップ6.1/6.3でエージェント設定が完了したら、ステップ2.6に戻って詳細設計を継続する
  - `/idea` のペルソナで十分なケースも多い。専用ペルソナが必要かどうかはユーザーと相談して決める
  - エージェントの紐づけは「どのタスク（ステップ）のことか」を必ず明示してから進むこと
